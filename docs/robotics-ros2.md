# padlock for Robotics — ROS 2 and Cortex-M

Robotics software runs on two very different hardware profiles:

- **ARM Cortex-M** (STM32, NXP, Nordic): microcontrollers driving actuators and sensors — memory-constrained, often no cache on M0/M3, 32-byte cache lines on M4/M7.
- **ARM Cortex-A / x86** (Raspberry Pi, Jetson, Intel NUC): SBCs running ROS 2 nodes — standard cache-line concerns, false sharing between DDS threads.

padlock supports both via the `--target` flag.

---

## Target triples for embedded ARM

| MCU family | Rust triple | padlock target | Cache line |
|---|---|---|---|
| Cortex-M0 / M0+ | `thumbv6m-none-eabi` | `cortex_m` | none |
| Cortex-M3 | `thumbv7m-none-eabi` | `cortex_m` | none |
| Cortex-M4 / M4F | `thumbv7em-none-eabi` | `cortex_m4` | 32 bytes |
| Cortex-M7 | `thumbv7em-none-eabihf` | `cortex_m4` | 32 bytes |
| Cortex-M33 | `thumbv8m.main-none-eabi` | `cortex_m4` | 32 bytes |

```bash
# Analyse firmware source for an STM32F4 (M4, 32-byte cache line)
padlock analyze firmware/src/ --target thumbv7em-none-eabi

# Analyse an Arduino-compatible MCU (M0, no cache — false sharing suppressed)
padlock analyze firmware/src/ --target thumbv6m-none-eabi
```

For `cortex_m` targets (no cache), padlock suppresses `FalseSharing` and `LocalityIssue` findings automatically — these findings are meaningless without a cache. `PaddingWaste` and `ReorderSuggestion` still apply and are especially valuable on memory-constrained chips.

## Example: sensor fusion struct on STM32F4

```c
// firmware/sensors/imu.h
typedef struct {
    bool    calibrated;     // 1 byte
    // 3 bytes padding
    float   accel_x;        // 4 bytes (4-byte aligned)
    float   accel_y;        // 4 bytes
    float   accel_z;        // 4 bytes
    bool    overflow;       // 1 byte
    // 3 bytes padding
    float   gyro_x;         // 4 bytes
    float   gyro_y;         // 4 bytes
    float   gyro_z;         // 4 bytes
} ImuData;                  // total: 32 bytes — 6 wasted (19%)
```

```
$ padlock analyze firmware/sensors/imu.h --target thumbv7em-none-eabi

[✗] ImuData  32B  fields=8  holes=2  score=52
    [MEDIUM] Padding waste: 6B (19%) — 3B after `calibrated` (offset 1), 3B after `overflow` (offset 17)
    [HIGH]   Reorder fields: 32B → 28B (saves 4B): accel_x, accel_y, accel_z, gyro_x, gyro_y, gyro_z, calibrated, overflow
```

On an STM32F4 with 192KB SRAM, saving 4 bytes per `ImuData` instance lets you store 50% more samples in a ring buffer before overflowing to flash.

Fix:

```c
typedef struct {
    float   accel_x;        // offset  0
    float   accel_y;        // offset  4
    float   accel_z;        // offset  8
    float   gyro_x;         // offset 12
    float   gyro_y;         // offset 16
    float   gyro_z;         // offset 20
    bool    calibrated;     // offset 24
    bool    overflow;       // offset 25
    // 2 bytes trailing padding
} ImuData;                  // total: 28 bytes — 2 wasted (7%)
```

Apply automatically:

```bash
padlock fix firmware/sensors/imu.h
```

## ROS 2 — analysing generated message structs

ROS 2 generates C++ message structs from `.msg` files via `rosidl`. The generated structs are not layout-optimised — the generator preserves field declaration order from the `.msg` file, which is written for human readability, not cache efficiency.

```bash
# After building your workspace:
padlock analyze install/my_pkg/include/my_pkg/msg/ --sort-by waste

# Or analyze the installed headers directly
padlock analyze /opt/ros/humble/include/sensor_msgs/msg/ --sort-by waste
```

### Example: `sensor_msgs/PointCloud2`

The `PointCloud2` message header has several small boolean fields mixed with larger integer fields:

```cpp
// Generated by rosidl — do not edit directly
struct PointCloud2 {
    std_msgs::msg::Header  header;   // large nested struct
    uint32_t               height;
    uint32_t               width;
    // ... field descriptors ...
    bool                   is_bigendian;  // 1 byte
    // 3 bytes padding
    uint32_t               point_step;
    uint32_t               row_step;
    // ... data vector ...
    bool                   is_dense;     // 1 byte
    // 7 bytes trailing padding
};
```

```
$ padlock analyze /opt/ros/humble/include/sensor_msgs/msg/point_cloud2.hpp \
    --filter PointCloud2

[✗] PointCloud2  ...B  fields=...  holes=2  score=...
    [LOW] Padding waste: 11B — 3B after `is_bigendian`, 7B after `is_dense`
    [LOW] Reorder fields: saves 8B
```

Since these are generated files, use `exclude_paths` in `.padlock.toml` to skip them from CI gating, and note the issues upstream to the rosidl team or your message layout.

## Cortex-M CI integration with `memory.x`

For embedded Rust firmware targeting Cortex-M, integrate padlock into your existing CI:

```yaml
# .github/workflows/firmware.yml
- name: Analyse struct layouts for STM32F4
  run: |
    cargo install padlock-cli
    padlock analyze src/ --target thumbv7em-none-eabi --fail-on-severity high
```

For C/C++ firmware using CMake or Make:

```yaml
- name: Analyse firmware struct layouts
  uses: gidotencate/padlock@v0.9.3
  with:
    path: firmware/Core/Inc/
    target: thumbv7em-none-eabi
    fail-on-severity: high
```

## Protocol structs for ROS 2 serialization (CDR)

ROS 2 uses CDR (Common Data Representation) serialization. CDR requires fields to be aligned to their natural size, meaning padding in your message struct translates directly to padding in the wire format — increasing bandwidth and serialization time.

padlock's `PaddingWaste` finding on a ROS 2 message struct is directly actionable: every byte of padding you eliminate is a byte that doesn't cross the DDS middleware on every publish/subscribe cycle.

Add a padding assertion to prevent regressions in performance-critical messages:

```rust
// Rust with padlock-macros
#[padlock::assert_no_padding]
#[repr(C)]
struct ImuReading {
    timestamp_ns: u64,   // 8 bytes at offset 0
    accel: [f32; 3],     // 12 bytes at offset 8
    gyro: [f32; 3],      // 12 bytes at offset 20
}                        // total: 32 bytes — no padding
```
